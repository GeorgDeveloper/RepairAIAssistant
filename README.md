# Repair AI Assistant

Система технической поддержки с искусственным интеллектом для анализа и решения проблем ремонта оборудования.

## Архитектура

- **assistant-core** (8080) - Основной API и бизнес-логика
- **assistant-web** (8081) - Веб-интерфейс
- **assistant-telegram** (8082) - Telegram бот
- **assistant-base_update** (8084) - Синхронизация данных SQL Server → MySQL (метрики каждые 3 минуты, перенос нарядов BD по расписанию и вручную)

## Быстрый старт

## Ручной перенос нарядов BD (assistant-base_update)

Модуль забирает данные из SQL Server (`REP_BreakdownReport` в CMMS) и пишет в MySQL таблицу `equipment_maintenance_records`.

### Автоматический режим

- Ежедневно в **08:02** по часовому поясу **Europe/Moscow** запускается перенос за интервал **производственных суток**: с **вчера 08:00** до **сегодня 08:00** (те же границы, что и у ручного догруза по дням).
- После вставки выполняется заполнение производных полей (`cause`, `staff`, `production_day` и др.) и удаление строк по правилам фильтрации (ложные вызовы, Tag, «в исполнении» и т.д. — см. `DataTransferService`).

### HTTP API (порт **8084**, базовый путь `/api/transfer`)

Все методы — **POST**.

| Путь | Назначение | Параметры |
|------|------------|-----------|
| `/bd` | Перенос BD за **один** интервал | Необязательно: `start`, `end` в формате ISO-8601 (`2026-04-04T08:00:00`). Если не заданы — как у cron: вчера 08:00 — сегодня 08:00 по Москве. |
| `/bd/backfill` | Догруз **нескольких** производственных дней подряд | Обязательно: `from`, `to` — даты **`YYYY-MM-DD`** (календарный день начала смены). Для каждого дня D выполняется окно **[D 08:00, D+1 08:00)** по Москве. |
| `/tag` | Перенос Tag за один интервал | Аналогично `/bd` (`start`, `end`). |
| `/tag/backfill` | Догруз Tag за диапазон дней | Как `/bd/backfill` (`from`, `to`). |

Параметры можно передавать **в query** (`?from=...&to=...`) или **в теле** POST как `application/x-www-form-urlencoded` (несколько `-d` у `curl`).

### Важно: Windows и символ `&`

В **cmd.exe** символ **`&`** — разделитель команд. Если URL не заключён в кавычки, оболочка обрезает запрос на первом `&`, вторую часть пытается выполнить как команду (сообщение вроде `'to' is not recognized...`), а сервер получает **не все** параметры → часто **HTTP 400** (`to` / `end` отсутствует).

Для **`/bd`** при отсутствии `start` или `end` сервер **намеренно** подставляет интервал «вчера 08:00 — сегодня 08:00» по Москве — поэтому кажется, что «игнорируется» заданный диапазон: до сервера дошёл только `start`, а `end` потерялся из‑за `&`.

**Рабочие примеры для Windows (cmd):** весь URL в **двойных кавычках**:

```bat
curl -X POST "http://10.130.143.200:8084/api/transfer/bd/backfill?from=2026-04-04&to=2026-04-05"
curl -X POST "http://10.130.143.200:8084/api/transfer/bd?start=2026-04-04T08:00:00&end=2026-04-05T08:00:00"
```

Либо **без `&` в строке для cmd** — параметры в теле запроса:

```bat
curl -X POST http://10.130.143.200:8084/api/transfer/bd/backfill -d from=2026-04-04 -d to=2026-04-05
curl -X POST http://10.130.143.200:8084/api/transfer/bd -d start=2026-04-04T08:00:00 -d end=2026-04-05T08:00:00
```

**PowerShell:** либо кавычки вокруг URI, либо явно `curl.exe` (чтобы не сработал алиас), например:

```powershell
curl.exe -X POST "http://10.130.143.200:8084/api/transfer/bd/backfill?from=2026-04-04&to=2026-04-05"
Invoke-RestMethod -Method Post -Uri "http://10.130.143.200:8084/api/transfer/bd/backfill?from=2026-04-04&to=2026-04-05"
```

### Дубликаты

Перед вставкой проверяется, есть ли в MySQL уже строка с тем же номером наряда **`code`** и тем же **`start_bd_t1`** / **`stop_bd_t4`** (включая сравнение с `NULL`). Совпадение — строка **не** вставляется повторно (удобно для повторного прогона догруза). У **разбитых** на части длинных нарядов у частей разные границы времени — каждая часть проверяется отдельно.

## Разработка

### Структура проекта
```
repair-ai-assistant/
├── application.yml          # Единая конфигурация
├── assistant-core/          # Основной модуль
├── assistant-web/           # Веб-интерфейс  
├── assistant-telegram/      # Telegram бот
├── assistant-base_update/   # Модуль синхронизации данных
└── target/                  # Сборка
```

### Сборка
```bash
mvn clean package
```
После сборки артефакты копируются скриптом `build.bat` в каталог `target`:

- WAR: `target/jars/assistant-*.war` (включая `assistant-base-update-0.0.1-SNAPSHOT.war`)
- Конфиги: `target/config/*-application.yml` (включая `base-update-application.yml`)
- Скрипты запуска/остановки: `target/start.bat`, `target/stop.bat`

### Тестирование
```bash
mvn test
```
